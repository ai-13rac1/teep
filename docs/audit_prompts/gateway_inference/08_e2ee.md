# Section 08 — E2EE: End-to-End Encryption & Trust Delegation

## Scope

Audit the end-to-end encryption mechanism that protects inference content from the gateway, the signing key binding and lifecycle, key exchange, symmetric encryption/decryption, and the trust delegation properties of the gateway inference architecture.

In the gateway inference model, the model backend's attestation provides an Ed25519 public key (`signing_key`) that is bound to the model backend's TDX quote via REPORTDATA. The proxy uses this key for X25519-based E2EE (converting Ed25519 to X25519 for key exchange), encrypting request messages so that even the gateway cannot read them, and decrypting response messages that were encrypted by the model backend.

> **Known divergence**: Venice uses a different E2EE protocol — secp256k1 ECDH + AES-256-GCM with a keccak256-derived REPORTDATA binding scheme. The Venice protocol is actually a previous version of the nearcloud E2EE protocol, and so this section focuses on the mechanics of the latest nearcloud protocol.

## Background: E2EE in the Gateway Architecture

The E2EE mechanism is a critical security feature unique to the gateway inference model. Without E2EE, a compromised gateway could read all inference traffic (prompts and responses). With E2EE:

```
Client → teep proxy ─── encrypted ──→ gateway CVM ─── encrypted ──→ model backend CVM
                    (gateway cannot decrypt)              (model backend decrypts)
```

The signing key from the model backend's attestation is the anchor for this scheme. The trust chain is:
1. Model backend TDX quote → REPORTDATA binds signing_key and nonce
2. `tdx_reportdata_binding` enforced factor verifies this binding
3. Ed25519-to-X25519 key conversion: the model's Ed25519 signing key is converted to X25519 for ECDH
4. Key exchange: proxy generates fresh ephemeral Ed25519 key pair, derives X25519 private key from Ed25519 seed
5. ECDH shared secret → HKDF-SHA256 with info string `"ed25519_encryption"` → XChaCha20-Poly1305 key
6. Request content encrypted with XChaCha20-Poly1305 before reaching the gateway
7. Response content decrypted by the proxy after passing through the gateway

## Required Checks

### Signing Key Validation

Verify and report:
- that the `signing_key` is obtained from the model backend's attestation (not the gateway's attestation),
- that the `signing_key` is present in the attestation response — the `signing_key_present` factor MUST be enforced,
- that the key is validated as a 64-hex-character Ed25519 public key (32 bytes),
- that the key is bound to the model backend's TDX quote via `tdx_reportdata_binding` — without this binding, a MITM could substitute the key,
- that E2EE is only activated when `tdx_reportdata_binding` has passed — if binding fails, E2EE MUST be refused (not silently degraded to plaintext),
- behavior when `signing_key` is absent (expected: `signing_key_present` factor fails, blocking the request if enforced).

### Key Exchange

Verify and report:
- that a fresh ephemeral Ed25519 key pair is generated per request (not reused across requests),
- that the Ed25519 seed (first 32 bytes of the 64-byte private key) is used to derive the X25519 private key via SHA-512 clamping (matching `crypto/ed25519` to `crypto/ecdh` X25519 conversion),
- that the model's Ed25519 public key is converted to X25519 using `edwards25519.SetBytes` and Montgomery conversion,
- that the ECDH shared secret is derived from the X25519 key pair,
- that the shared secret is used as input to HKDF-SHA256 with info string `"ed25519_encryption"`,
- the HKDF parameters: hash algorithm (SHA-256), salt value (if any), info string context,
- that the derived key length matches XChaCha20-Poly1305 requirements (32 bytes),
- that the ephemeral Ed25519 public key (64 hex chars) is transmitted to the model backend via the `X-Client-Pub-Key` HTTP header,
- that the `X-Signing-Algo: ed25519` and `X-Encryption-Version: 2` headers are set.

### Symmetric Encryption/Decryption

Verify and report:
- that XChaCha20-Poly1305 is used for symmetric encryption/decryption (not AES-GCM — this is the nearcloud protocol; Venice uses AES-256-GCM),
- that random nonces are generated per encryption operation (not reused),
- nonce size (expected: 24 bytes for XChaCha20-Poly1305),
- that nonce generation uses `crypto/rand` and that nonce failure is a hard error,
- that the authentication tag is verified during decryption (XChaCha20-Poly1305 provides AEAD — verify tag verification is not skipped),
- that decryption failure is a hard error (not silently degraded to plaintext),
- whether additional authenticated data (AAD) is used in the AEAD construction.

### Key Material Lifecycle

Verify and report:
- that the session private key is zeroed after use (`Session.Zero()` or equivalent),
- that zeroing uses explicit byte-level clearing (`for i := range key { key[i] = 0 }`), acknowledging Go GC limitations,
- that the zeroing is in a `defer` block to ensure it executes even on error paths,
- that no copies of the private key material are retained in log buffers, error messages, or cache structures.

### E2EE Header Protection

A key security benefit of the gateway inference model is that E2EE protects request and response content from the gateway. Verify and report:
- that request message content is encrypted before being sent through the gateway,
- that response message content (both streaming SSE chunks and non-streaming JSON bodies) is decrypted by the proxy,
- that non-encrypted content fields in an E2EE session are treated as errors (not silently accepted as plaintext),
- that the "role" and "refusal" fields are correctly exempted from encryption expectations (these are in the `NonEncryptedFields` allowlist in `internal/e2ee/relay.go`),
- that streaming SSE decryption handles all encrypted delta fields (content, reasoning_content, etc.), not just a hardcoded subset,
- that error responses from the model (which may not be encrypted) are handled gracefully without leaking session state.

### Signing Key Cache

The signing key (model backend's public key) MAY be cached with a short TTL. Verify and report:
- that the signing key cache has a shorter TTL than the attestation report cache,
- that the signing key is only cached after successful REPORTDATA binding verification,
- that a key rotation (different signing key from the same provider/model) emits a warning,
- that the cached signing key is the one verified by REPORTDATA binding, not from a subsequent unverified response,
- cache invalidation behavior on REPORTDATA binding failure.

### Trust Delegation and Gateway Compromise Resilience

The gateway inference model introduces a trust delegation that does not exist in the direct inference model. The proxy trusts:
1. the gateway's TLS certificate (pinned via gateway attestation REPORTDATA),
2. the gateway to faithfully forward the model attestation response,
3. the model backend's signing key (bound via model REPORTDATA).

The audit MUST evaluate the security properties that survive a gateway compromise:

- **E2EE key integrity**: Because the model backend's signing key is bound to the model's TDX quote via REPORTDATA (not to the gateway's quote), a compromised gateway cannot substitute the E2EE key without also compromising the model backend's TDX quote. Verify this binding chain.

- **Request/response confidentiality**: If E2EE is active, the gateway sees only encrypted content. A compromised gateway cannot read or modify encrypted message content. Verify that no plaintext content leaks through HTTP headers or other side channels.

- **Metadata visibility**: A compromised gateway can observe HTTP headers, request timing, model selection, and response sizes — even with E2EE enabled. This is a residual risk inherent to the gateway architecture. Document what metadata is visible.

- **Attestation relay integrity**: A compromised gateway could attempt to relay a different model backend's attestation (pointing to a compromised machine). The audit MUST document whether there is a binding between the gateway and the specific model backend that prevents the gateway from routing to an unattested machine.

The audit MUST quantify the residual risk and clearly state which attack scenarios are mitigated by E2EE and which are not.

### e2ee_usable Factor

The `e2ee_usable` factor tests live E2EE encryption/decryption. In the proxy path, the factor starts as Skip (pending live test) with `Deferred: true`, which exempts it from `BuildReport`'s Skip→Fail promotion even when enforced. This solves the chicken-and-egg problem: the proxy can forward the first E2EE request without blocking, then promotes the factor to Pass via `MarkE2EEUsable` after a successful relay. Post-relay decryption failures trigger `MarkE2EEFailed` (demoting to Fail), `e2eeFailed` blocking, and cache invalidation — fail-closed for all subsequent requests until re-attestation. In the `teep verify` path, `E2EETest` is populated directly and `Deferred` is false, so the factor evaluates normally.

## Go Best-Practice Audit Points

- **`defer` for key zeroing**: Verify that `Session.Zero()` is called in a `defer` immediately after key creation.
- **Error handling in crypto paths**: Every cryptographic operation (ECDH, HKDF, AEAD seal/open) must error-check and fail closed.
- **No `panic` on crypto failure**: Encryption/decryption failures should return errors, not panic.
- **`crypto/rand` for nonces**: XChaCha20-Poly1305 nonces (24 bytes) must come from `crypto/rand`.
- **Struct field visibility**: Verify that the ephemeral private key is not exported (lowercase field name) to prevent accidental external access.

## Cryptography Best-Practice Audit Points

- **Ed25519-to-X25519 conversion correctness**: Verify that the Ed25519 public key is correctly converted to X25519 via Montgomery form. Verify that the Ed25519 seed is correctly clamped for X25519 private key derivation (SHA-512 of seed, clamp bits 0, 1, 2, 253, 254, 255).
- **HKDF parameters**: Verify HKDF uses SHA-256, and that the info string `"ed25519_encryption"` is unique and context-bound (prevents cross-protocol key reuse).
- **XChaCha20-Poly1305 nonce uniqueness**: With random 24-byte nonces, the collision probability is negligible even for very large numbers of encryptions. Verify that nonce generation uses `crypto/rand` exclusively.
- **No key reuse across sessions**: Each E2EE session MUST use a fresh ephemeral Ed25519 key pair. Verify that session keys are never cached or reused across multiple inference requests.
- **Authenticated encryption**: XChaCha20-Poly1305 provides both confidentiality and integrity. Verify that the authentication tag is always checked before processing decrypted content.

## General Security Audit Points

- **Downgrade to plaintext**: If E2EE fails (key exchange error, decryption error), the proxy MUST NOT fall back to forwarding plaintext. Verify that all E2EE failure modes result in request rejection.
- **Gateway cannot forge E2EE**: A compromised gateway cannot decrypt the content because it does not have the model backend's private key. Verify this by tracing the key material derivation path.
- **Replay protection**: Each E2EE session uses a fresh ephemeral key pair. Verify that a replayed request (with the same encrypted content) cannot be accepted by a different session.
- **Side-channel leakage**: Even with E2EE, the gateway can observe request/response sizes, timing, and HTTP headers. Document what information is leaked through these side channels.

## Known Divergence: Chutes/Sek8s E2EE

Chutes uses a completely different E2EE protocol based on ML-KEM-768 (post-quantum key encapsulation) rather than Ed25519/X25519/XChaCha20-Poly1305.

### Chutes E2EE Protocol

- **Key encapsulation**: ML-KEM-768 (FIPS 203). The server's ML-KEM-768 public key is returned as base64 in the instances endpoint response.
- **Key exchange / wire format**: Client encapsulates a shared secret using the server's ML-KEM-768 public key. The resulting KEM ciphertext is part of the Chutes E2EE wire format implemented in `internal/e2ee/chutes.go` / `internal/e2ee/relay_chutes.go`; it is **not** sent as a standalone `X-E2E-Ciphertext` header.
- **Key derivation**: HKDF-SHA256 with only the first 16 bytes of the KEM ciphertext as salt and distinct info strings:
  - `"e2e-req-v1"` for request encryption key
  - `"e2e-resp-v1"` for non-streaming response key
  - `"e2e-stream-v1"` for streaming response key
- **Symmetric cipher**: ChaCha20-Poly1305 (not XChaCha20) with 12-byte random nonces.
- **REPORTDATA binding**: The ML-KEM-768 public key is bound to the TDX quote via `SHA256(nonce_hex + pubkey_base64)` in `REPORTDATA[0:32]`.

### Chutes Streaming Protocol

Chutes uses custom SSE event types for encrypted streaming:
- `e2e_init`: Contains the encrypted request acknowledgment.
- `e2e`: Contains encrypted response chunks.
- Standard `data:` SSE events are not expected within an E2EE chutes session.

### Key Audit Differences

- No Ed25519-to-X25519 conversion (ML-KEM-768 is a KEM, not a DH scheme).
- No signing key concept — the server provides a KEM public key, not a signing key.
- `signing_key_present` factor is in `ChutesDefaultAllowFail`.
- `e2ee_usable` factor is in `ChutesDefaultAllowFail`.
- `e2ee_capable` is enforced (the instances endpoint must return valid ML-KEM-768 keys).
- Key material zeroing applies to the KEM shared secret and derived session keys.
- Gateway compromise resilience analysis differs: the Chutes gateway is unattested, so E2EE compensates for the lack of attestation-bound TLS pinning. The audit SHOULD verify that the E2EE binding to the backend instance's TDX quote is sufficient to protect against a compromised gateway routing to an unattested machine.
- No gateway TDX quote exists to independently verify the gateway.
